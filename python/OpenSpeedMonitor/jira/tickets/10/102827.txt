 Reaction Automatic-Interface-Copy zu 	"Funktionsweise:



Die Reaction „Automatic Interface Copy“ kopiert automatisch die Schnittstellen der Vorgänger, wenn ein Nachfolger (ein neues Informationssystem) angelegt wird. Es ist getriggert, wenn man in der Maske eines Informationssystems die Option ""Neues Release"" auswählt. Wenn der Informationssystem die Schnittstellen hat, werden diese auch für das neue Release kopiert.



Dokumentation:



Diese Reaction ist in iteraWiki unter iteraplan Reactions / Automatic-Interface-Copy dokumentiert:

https://iteraweb.iteratec.de/confluence/display/ITM/iteraplan+Reactions 



Code:



Der aktuelle Stand der Code ist in iteraGIT verfügbar (und auch als Anhang in diesem Email):

https://iteragit.iteratec.de/iteraplan-Plugin/iteraplan-Reactions/tree/master/Automatic-Interface-Copy 





Status:



Die Reaction ist Code-Technisch-Gesehen fertig (soweit es möglich war). Das Ausführen (zum Zwecke der Tests in einer nicht-produktiven Instanz) ist möglich, und die Schnittstellen sollen entsprechend kopiert werden. Nichtdestotrotz, es existiert ein Problem in iteraplan 5.4, der Fertigstellung der Reaction in dieser Version von interaplan verhindert.



Solange „Partial Elastic Update“ Problem in iteraplan (ein generisches Problem, dass Reactions API oder Import/Export beeinflusst) nicht beseitigt wird, funktioniert diese Reaction inkorrekt und kann zum Problemen mit Datenbasis führen! Daher ist aktuell der Einsatz dieser Reaction in produktiven Umgebungen definitiv abzusehen! Auch wenn es scheint, dass der Code der Reaction korrekt ist (soweit man es testen konnte), ist es ein schwerwiegendes Problem, der nur durch die Korrektur von Partial Elastic Update (und dementsprechend Reaction API) beseitigt sein kann.



Beachte: Das Problem ist erstmals schwierig zu merken. Wenn man die Reaction erstmals nutzt, scheint alles zu funktionieren. Nichtdestotrotz schon beim Kopieren der ersten Interface erstellt es aktuell (iteraplan 5.4) zwei Information Flows anstelle von einem (ein „richtiges“ und ein unnötiges, der ein negativer ID hat – verursacht durch den Problem in iteraplan selbst). Man kann es in iteraplan Client gar nicht merken, da Objekte mit negativer ID in iteraplan Interface nicht gezeigt werden (es ist aber z.B. in Excel-Export der Daten sichtbar). Nichtdestotrotz, schon beim ersten Nachfolger entsteht ein Problem in der Datenbasis. Dieser wird erst dann wirklich sichtbar, wenn man für so ein Nachfolger ein weiteres Nachfolger erstellen möchte – was zum Ausführungsfehler führen wird.
